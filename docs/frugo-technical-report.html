Section 01

Executive Summary

FruGo is a full-stack e-commerce web application built on Laravel 11 (PHP 8.2) for selling fresh fruits and vegetables online — a complete platform combining a customer-facing shop with a back-office administration system in a single application.

Three roles — Administrator, Product Manager, and Customer — each have distinct capabilities. Customers browse an approved product catalogue, add items to a session-based cart, and place orders, all authenticated via passwordless OTP login (a 6-digit code emailed to them). Admins and Product Managers use traditional email/password login and operate the back-office: creating products, managing approval workflows, and administering user accounts.

The application is containerised with Docker Compose, deployable with a single command, and structured around SOLID principles using Repository and Service design patterns.

Section 02

Introduction & Scope

Purpose

FruGo delivers a frictionless shopping experience for customers (no password to remember) while maintaining controlled product publishing — products created by Product Managers are only visible in the shop after explicit approval by an Administrator.

In Scope

  • Customer shop — product catalogue, search, category filter, product detail pages
  • Session cart — add, update quantity, remove, clear; persisted in PHP session
  • Checkout & order tracking — place order from cart, view order history and status
  • OTP passwordless auth — customers authenticate via 6-digit email code (10-minute expiry)
  • Admin / PM back-office — full CRUD product management, approval workflow
  • Admin user management — create staff accounts, assign roles
  • Email notifications — OTP delivery, product approved/rejected status emails
  • Docker Compose four-service containerised local environment

Out of Scope

  • Payment gateway integration
  • Real-time notifications or WebSockets
  • Product image file upload (managed via image path references)

Section 06

Module Descriptions

ShopController

Public product listing and detail. index() returns paginated approved products with search and category filter. No authentication required — the shop is publicly browsable.

CartController

Session-based cart stored as session('cart') = [productId => ['qty' => N, 'price' => X]]. Methods: add, update, remove, clear, index. Requires role:customer middleware — admins and managers cannot use the cart.

OrderController

store() runs in a database transaction: creates the Order, writes all OrderItem rows with snapshotted prices, then clears the session cart atomically. index() and show() serve order history and detail to the authenticated customer.

OtpLoginController

Full passwordless OTP flow — generates a 6-digit code, bcrypt-hashes it before storage, sends via email, verifies with Hash::check(), and deletes the token on success. See Section 8 for the complete flow.

ProductController

RESTful product management for admin and product_manager roles. Delegates business logic to ProductService. Additional actions beyond CRUD: approve(), reject(), submit(). Sends email notifications on approval and rejection.

Admin\UserController

Admin-only staff account management. Customer accounts are auto-created by OtpLoginController on first login — this controller only creates and manages admin and product_manager accounts.

Section 08

OTP Login Flow

Step 1 — Request OTP
  Customer enters email → POST /login/otp
  → Find or auto-create User with role=customer
  → Generate random 6-digit code
  → bcrypt hash → store in otp_tokens(email, hash, expires_at=now+10min)
  → Mail::to($email)->send(new OtpMail($code))
  → Redirect to verify form

Step 2 — Verify OTP
  Customer enters code → POST /login/otp/verify
  → Lookup token by email
  → Check expires_at > now()
  → Hash::check($submitted, $storedHash)
  → DELETE token immediately  ← prevents replay attacks
  → Auth::login($user) → session()->regenerate()
  → Redirect to shop homepage

Resend
  POST /login/otp/resend → delete existing token → restart Step 1

Section 11

Challenges & Solutions

OTP Token — Hashed Storage

Problem: Storing a 6-digit OTP in plaintext means a DB breach exposes all pending tokens immediately.

Solution: bcrypt hash stored in DB; plaintext exists only in memory and in the email. Hash::check() performs constant-time verification.

Session Cart Without a Database Table

Problem: A DB-backed cart needs its own table, user association before checkout, and abandoned-cart cleanup jobs.

Solution: PHP session array keyed by product ID. Zero schema, automatically expires with session. Trade-off: cart lost on session expiry — persisting to DB after login is a natural future enhancement.

Price Snapshot on Order

Problem: Storing only a product FK in order_items means future price changes retroactively alter historical order totals.

Solution: unit_price copied from $product->price into order_items at checkout time. Historical orders are immutable.

Named Route Conflict — Two Login Methods

Problem: Laravel's auth middleware redirects to the named route login. FruGo has two login paths — the name must resolve to one primary entry point.

Solution: Password login is named login (Laravel's default expectation). The page includes a prominent link to the OTP form. All middleware redirects land on the password page, which offers OTP as an alternative.

Section 14

Conclusion

FruGo demonstrates a complete, production-structured Laravel application combining a customer-facing e-commerce shop with a staff back-office — differentiated entirely by role-based access control within a single codebase.

The passwordless OTP authentication provides frictionless customer onboarding while bcrypt-hashed token storage and immediate post-use deletion meet the same security bar as traditional password authentication. Price-snapshotted orders and the product approval workflow form a coherent, trust-safe buying experience that can scale to real production use with the enhancements identified above.
