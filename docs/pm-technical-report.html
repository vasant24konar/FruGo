Section 01

Executive Summary

Project Manager is a back-office web application built on Laravel 11 (PHP 8.2) that enables organisations to manage product catalogues through a structured multi-role approval workflow. The system enforces a three-tier access model — Administrator, Product Manager, and Customer — with each role restricted to its own set of operations via middleware-enforced RBAC.

Products created by Product Managers enter a pending state and are only published after explicit approval by an Administrator. Product images are uploaded directly through the form and stored on the server — no manual file path entry is required. The application is containerised with Docker Compose (four services: Nginx, PHP-FPM, MySQL 8, MailHog), deployable with a single command, and structured around the SOLID principles using the Repository and Service design patterns.

This report covers the architectural decisions, database schema, module design, security controls, deployment infrastructure, and outstanding improvement opportunities for the system.

Section 02

Introduction & Scope

Purpose

Project Manager addresses the need for a controlled product publishing workflow where products require review and approval before they appear in a customer-facing store. It provides a centralised dashboard for staff accounts (admins and product managers) and is fully isolated from the public shop front.

In Scope

  • Multi-role user authentication — session-based login with email + password
  • Full CRUD product management with status tracking (pending / approved / rejected)
  • Product image upload — JPEG / PNG / WebP / GIF up to 2 MB, stored in public/img/products/
  • Admin product approval / rejection workflow with optional rejection reason
  • Admin user account management (create, assign role, deactivate)
  • Full-text product search across title and description
  • Email notifications via MailHog (dev) or SMTP (production)
  • Docker Compose four-service containerised environment
  • PSR-4 autoloading and PSR-12 code style enforced by Laravel Pint

Out of Scope

  • Customer-facing shop UI (separate FruGo application)
  • OTP / passwordless login (FruGo feature only)
  • Payment processing
  • Real-time notifications

Table Descriptions

TablePurposeKey Columns
usersAll system users. Single table with role discriminator column — no separate tables per role.role (enum), email (unique), password (bcrypt)

Image Storage

Uploaded product images are stored in public/img/products/ with a uniqid()-generated filename to prevent collisions and path guessing. The image_path column stores only the relative path (e.g. img/products/product_64a3f1.jpg), which is passed directly to Laravel's asset() helper in Blade templates.

Migration Strategy

Migrations are version-controlled alongside the source code. deploy.sh runs php artisan migrate --force on each deployment. make fresh runs migrate:fresh --seed for development resets. Seeders create one admin and one product_manager account with known credentials for demonstration.

Indexes

  • products(created_by) — speeds up "list my products" queries for product managers
  • products(status) — speeds up the pending approval dashboard filter
  • FULLTEXT(products.title, products.description) — supports keyword search without full table scans

Section 06

Module Descriptions

ProductController

RESTful resource controller. Validates via ProductRequest, delegates business logic to ProductService, returns views or redirects. Two additional methods beyond standard CRUD handle the approval workflow: approve() and reject() mapped to POST /products/{id}/approve and POST /products/{id}/reject.

Image upload handling lives in a private storeImage(UploadedFile $file) helper on the controller. It uses File::ensureDirectoryExists() to create public/img/products/ on first use, moves the file with a unique name, and returns the relative path string. The controller replaces the UploadedFile object from $request->validated() with this string before passing the data to the service layer — keeping the service and repository free from file-system concerns.

app/Http/Controllers/ProductController.php — storeImage helper

private function storeImage(UploadedFile $file): string
{
    $dir = public_path('img/products');
    File::ensureDirectoryExists($dir);
    $filename = uniqid('product_', true) . '.' . $file->getClientOriginalExtension();
    $file->move($dir, $filename);
    return 'img/products/' . $filename;
}

ProductService

Owns all business rules. createProduct() forces status = pending for product managers and status = approved for admins. canModify() enforces ownership — a product manager may only edit their own products; an admin may edit any.

ProductRepository

Implements ProductRepositoryInterface. Uses eager loading (with('creator:id,name')) to avoid N+1 queries. All methods return typed values — no raw query results leak to upper layers.

Admin\UserController

Manages staff accounts. Reachable only via role:admin middleware. Provides paginated user listing, create (with role assignment), update (role change), and soft-deactivate. Customers cannot be created here — they are created only through OTP login in the FruGo application.

CheckRole Middleware

Reads the variadic $roles argument from the route definition (e.g. role:admin,product_manager), checks auth()->user()->hasAnyRole($roles), and aborts with HTTP 403 if the user's role is not in the allowed list.

RepositoryServiceProvider

Binds ProductRepositoryInterface to ProductRepository in the service container. Controllers depend only on the interface — the concrete class is named in exactly one place.

Section 07

Authentication & RBAC

Authentication Flow

POST /login
  → LoginRequest validates email + password format
  → RateLimiter: max 5 attempts per email/IP per minute
  → Auth::attempt(['email' => $email, 'password' => $password])
    → bcrypt verify against users.password
  → On success: session()->regenerate()  ← prevents session fixation
  → Redirect to /dashboard
  → On failure: back() with throttled error message

Three-Layer Role Enforcement

  1. Route middleware — role:admin / role:admin,product_manager on route groups. Unauthenticated and wrong-role requests never reach a controller.
  2. Form Request authorize() — ProductRequest::authorize() re-checks authentication before any data is validated, preventing forged requests that bypass the route layer.
  3. Service-layer ownership — ProductService::canModify($user, $product) prevents horizontal privilege escalation between product managers trying to edit each other's products.

Section 10

Challenges & Solutions

Storage Permissions in Docker

Problem: Files created on the Windows host inside storage/ lost write access for Docker's PHP-FPM process (www-data), causing 500 errors: Failed to open stream: Permission denied.

Solution: deploy.sh runs chown -R www-data:www-data /var/www/storage /var/www/bootstrap/cache /var/www/public/img/products after container start. Documented as a Makefile target: make permissions.

Image Upload — File vs. Path Validation

Problem: The original form used a text input for image_path, requiring users to manually type a relative file path. This was error-prone, required files to be pre-placed on the server, and accepted any string value as a "path".

Solution: Changed the field to a file input with enctype="multipart/form-data". The ProductRequest now uses Laravel's image rule (validates MIME type from the file stream, not just the extension) with an explicit mimes allowlist and a 2 MB cap. The controller's storeImage() helper writes the file and returns a path string, keeping the service and repository layers oblivious to file-system operations.

Shared Layout — PM vs. FruGo

Problem: When the two projects were separated into distinct Git branches, layouts/shop.blade.php was initially deleted from the PM branch under the assumption it was FruGo-specific. However, all admin views (dashboard, products, admin/users) extended it, causing a 500 error on login.

Solution: Retained layouts/shop.blade.php on the PM branch but rewrote it with Project Manager branding, shop route references removed, and a :root { --primary: #86B817 } declaration added so all views that reference the CSS variable render buttons and accents correctly.

FULLTEXT Search Minimum Word Length

Problem: MySQL FULLTEXT ignores words shorter than ft_min_word_len (default 4). Searching for "fig" returned no results.

Solution: Added a LIKE fallback in ProductRepository::search() for queries under 4 characters.

Section 13

Conclusion

Project Manager demonstrates a production-grade Laravel application structured around industry-standard design patterns (Repository, Service Layer, Dependency Injection) with security, maintainability, and deployability as primary concerns.

The three-role RBAC system, multi-layer access control, product approval workflow, and file upload capability provide a solid foundation for managing any catalogue requiring editorial oversight before public publication. Image uploads are validated at the MIME level, stored with unique filenames, and served directly by Nginx — with the controller layer acting as the only point of contact between the HTTP request and the file system.

The single-command Docker deployment ensures the environment is fully reproducible. The identified future enhancements — orphaned file cleanup, automated tests, cloud storage, and a CI/CD pipeline — represent natural next steps toward a fully production-hardened system.
